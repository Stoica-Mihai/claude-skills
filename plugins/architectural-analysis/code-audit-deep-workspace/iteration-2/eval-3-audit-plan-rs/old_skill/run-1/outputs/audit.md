**Perf (6)**
- `plan.rs:140` — `regex_convergence_check` uses `find_iter(...).count()`, scanning the entire post-image even when one match suffices. Fix: replace with `regex.is_match(after)` returning `0` or `1`; only the zero/non-zero distinction is used downstream.
- `plan.rs:167-170` — scripted convergence re-runs `rewrite_text_scripted` over the full post-image (re-executing the Rhai callback on every match) and throws the result away. Fix: do a cheaper `regex.is_match(after)` probe — if zero matches, no script can fire, so it's automatically convergent for the second pass.
- `plan.rs:120,159` — `scan` collects every walked path into a sorted `Vec<PathBuf>` before any rewrite work begins, even though the only consumer is `par_iter()` which neither needs ordering nor full materialization. Fix: have `walk_paths` return a parallel iterator (or drop the `sort()` in `walker.rs:88` when called from here) and stream into `par_bridge` so walking and rewriting overlap.
- `plan.rs:297,306` — `read_text_or_skip_binary` issues `fs::metadata` then `fs::read_to_string`, doubling syscalls per file (stat + open+stat+read). Fix: `File::open` once, call `metadata()` on the handle for size + perms, then `read_to_string` from the handle.
- `plan.rs:278` — `unified_diff` runs `similar::TextDiff::from_lines` for every changed file inside the rayon map, even though the diff is only ever consumed by `--diff` mode (it's unused in `--apply` and `--check`). Fix: thread the output mode into `PlanOptions` and skip diff rendering when not needed; pass a closure or `enum DiffMode { Render, Skip }`.
- `plan.rs:123-134,162-175` — `Vec<Result<Option<FileChange>>>` is fully materialized before `collect_changes` walks it sequentially, allocating one `Result<Option<_>>` slot per scanned file (most are `Ok(None)`). Fix: use `try_fold` + `try_reduce` (or `filter_map` + `collect::<Result<Vec<_>, _>>()`) to keep only the changed files in memory.

**Memory (4)**
- `plan.rs:73-81` — `FileChange` holds `after` (full post-image) *and* `diff` (unified diff with both before/after lines + context). The two carry the same change information; for any file that changed substantially, total footprint is ~2x the file size. Fix: store only `after` and render `diff` lazily on demand, or store only `diff` and reconstruct `after` for the commit phase from the pre-image (since pre-image is dropped) — pick one consumer per mode.
- `plan.rs:123` — `let results: Vec<Result<Option<FileChange>>>` keeps every per-file outcome (and every Error allocation) alive for the full duration of the scan. For a 1000-file walk with one Err mid-scan, all 1000 slots stay in memory until the sequential `collect_changes` reduces them. Fix: short-circuit on first error via `try_for_each_init` / `try_reduce` so the buffer never grows past the changed-files count.
- `plan.rs:264,278` — `before` (the full pre-image `String`) is held alive until `unified_diff` returns, while `outcome.after` (a second full-file `String`) sits next to it; peak per-worker memory is ~3x file size (`before` + `after` + `diff`). Fix: render `diff` first, drop `before` with `mem::drop`, and only then build the `FileChange` — or merge with the diff-laziness fix above so `before` is dropped immediately after `outcome.after != before` check passes.
- `plan.rs:80` — `permissions: Option<Permissions>` is always `Some(...)` at construction (line 284) but the type allows `None`. The `Option` wrapper adds a discriminant and forces a `match`/`if let` in commit.rs. Fix: drop the `Option` — the planner is the only constructor and it always populates it; use `Permissions` directly.

**Correctness (2)**
- `plan.rs:297` — `fs::metadata(path)` follows symlinks, but `walk_paths` defaults to *not* following symlinks (`follow_links(opts.follow_symlinks)` in `walker.rs:53`). If a symlink slips through (e.g. via `--type` filter on a target file the user didn't intend to follow), this re-stat happily traverses it, mismatching walk policy. Fix: use `fs::symlink_metadata` when `opts.walk_options.follow_symlinks` is false, or thread the policy through.
- `plan.rs:265` — `outcome.after == before` is a full string compare run *after* a `replace_all` that already knows whether substitutions happened. For convergent patterns with capture-only replacements (`(\w+)` → `$1`), every file allocates a fresh `after`, then a full byte-compare runs to discover nothing changed. Fix: have `rewrite_text` return a `Cow<str>` and treat `Cow::Borrowed` as "no change" without comparing — `regex::replace_all` already returns `Cow`, the `into_owned()` at `rewrite.rs:43` is what discards that signal.

**Complexity (2)**
- `plan.rs:248-286` — `process_one` is generic over two `Fn` closures (`R`, `C`) just so two callers can share it; the two callers (`plan_rewrite` line 130, `plan_rewrite_scripted` line 167) each instantiate a fresh monomorphization, doubling generated code without sharing call sites. Fix: pass an `enum Mode<'a> { Regex, Script(&'a mut ScriptWorker) }` and branch internally, or accept a single `dyn Fn` trait object — the per-file cost dwarfs an indirect call.
- `plan.rs:112-137,151-181` — `plan_rewrite` and `plan_rewrite_scripted` are 95% the same code, differing only in the rewrite/convergence closures and the `convergent_or_scripted` argument. Fix: collapse to one `plan_rewrite_with(mode)` that takes the mode enum above; the public API can keep both names as thin shims.

**Coupling (1)**
- `plan.rs:80 + commit.rs:143,155` — `FileChange.permissions` exists solely so `commit.rs::stage_one` can skip a `fs::metadata` call; it is set at one site (line 284), read at one site, and `Option`-wrapped in a way no other consumer benefits from. The "preserve mode" responsibility is leaking from commit into plan. Fix: have commit re-stat (one syscall per file at apply time is negligible vs. the rename + fsync it already does), or make the field `Permissions` (not `Option`) and document plan as the sole owner of metadata capture — pick one home.

**Top targets**
1. `plan.rs:73-81` + `rewrite.rs:43` — collapse the dual `after`/`diff` storage and lift `Cow` through `rewrite_text` to (a) halve per-file memory and (b) skip the redundant `outcome.after == before` compare. Highest leverage: it touches memory shape *and* perf in one change.
2. `plan.rs:140,167-170` — replace both convergence checks with `regex.is_match(after)`. Eliminates a second full-file regex sweep and a full second script execution per changed file. Trivial diff, large per-file speedup on big files.
3. `plan.rs:120,159` + `walker.rs:88` — stop materializing + sorting the full file list before parallel rewrite; overlap walking with rewriting via `par_bridge`. Wins on cold-cache large trees where the walk currently blocks all workers.
4. `plan.rs:112-181` — fold `plan_rewrite` and `plan_rewrite_scripted` into one orchestrator with a `Mode` enum, drop the dual generic monomorphizations in `process_one`. Removes the structural duplication that is currently the strongest pressure to keep memory/perf fixes consistent across two near-identical paths.
