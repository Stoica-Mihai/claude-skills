**Perf (5)**
- `plan.rs:139-141` — regex convergence re-scans the entire post-image of every changed file even though `CompiledPattern::is_convergent()` already proves the static property once. Fix: skip the per-file scan when `compiled.is_convergent()` is true; gate `regex_convergence_check` behind `!compiled.is_convergent() && !opts.allow_non_convergent`.
- `plan.rs:168-171` — scripted convergence check calls `rewrite_text_scripted` a second time on `outcome.after`, re-running every Rhai callback (often the per-file hot path) just to count residual matches. Fix: count residual matches with `pattern.regex().find_iter(&outcome.after).count()` and skip the re-eval; only re-eval when that count is non-zero.
- `plan.rs:265` — `outcome.after == before` does a full byte-compare of the whole file on the post-rewrite path. Fix: rely on `outcome.matches == 0` to short-circuit; only fall back to the string compare when matches > 0 (rare degenerate case), or have `rewrite_text` return an `Option<String>` so the no-change case never allocates `after` at all.
- `plan.rs:123-134` / `162-175` — `par_iter().collect()` materialises a `Vec<Result<Option<FileChange>>>` of size N before any filtering; every kept `after` is double-wrapped in `Some(Ok(..))` until `collect_changes` unwraps it. Fix: use `par_iter().filter_map(..).collect::<Result<Vec<_>>>()` (or `try_reduce`) so unwrapped `FileChange`s land directly in the final `Vec`.
- `plan.rs:297, 306` — `fs::metadata` then `fs::read_to_string` issues stat + open + read per file; the stat is only used for the size cap. Fix: `File::open` once, `file.metadata()` for the size check, then `read_to_string(&mut file)` — same call count but one path-resolution and no TOCTOU between size check and read.

**Correctness (2)**
- `plan.rs:208, 223` — `total_matches` only sums matches in *kept* changes; any file where `outcome.matches > 0` but `outcome.after == before` (no-op rewrite that hit matches) is dropped from the count fed to `check_match_counts`, so `--at-least`/`--at-most` see a smaller number than truly matched. Fix: sum matches from every `RewriteOutcome` (including skipped ones) and pass that to the guard; keep the `changes` vec for emission.
- `plan.rs:297-304` — size check on `metadata()` followed by a separate `read_to_string` opens a TOCTOU window where the file can grow past `max_bytes` between the two syscalls. Fix: open first, then check `file.metadata().len()` against the cap before reading, so the size limit applies to the bytes you actually read.

**Memory (4)**
- `plan.rs:73-81` — `FileChange` holds both `after` (full post-image) and `diff` (unified diff encoding new relative to old), roughly doubling the planner's peak footprint per changed file; the apply phase only consumes `after`, the user only consumes `diff`. Fix: render the diff lazily (store `before` instead and build the diff on demand for printing) or stream diffs to the writer as files complete and keep only `after` in the plan.
- `plan.rs:135, 192-200` — `collect_changes` builds the final `Vec<FileChange>` after the par-iter has already built `Vec<Result<Option<FileChange>>>`; both live in memory at the same time, briefly holding 2x the per-file `String`s. Fix: collapse into a single `try_collect` over a `filter_map` so only one vector ever exists.
- `plan.rs:259-285` — `process_one` keeps `before` and `outcome.after` alive together for the lifetime of the diff render even after the post-image is final, so per-worker peak is `2 * filesize`; on a large file that doubles the rayon working set per thread. Fix: drop `before` immediately after `unified_diff` returns (rebind to `_`), or feed `unified_diff` a streaming line iterator so the `before` `String` can be dropped before `after` is moved into the `FileChange`.
- `plan.rs:80` — `permissions: Option<Permissions>` is always populated by the regex pipeline (`Some(permissions)` at line 284); the `Option` is dead weight everywhere except a hypothetical caller that bypasses `process_one`. Fix: make it `Permissions` once every constructor is in-tree, or drop the field and have `apply_changes` re-stat (the comment claims that's the cost being avoided — verify with a bench before dropping).

**Complexity (2)**
- `plan.rs:248-286` — `process_one` takes 5 parameters (pattern, path, opts, rewrite closure, convergence closure) and threads two distinct fallible operations through generics; the rewrite/convergence pair is really one trait. Fix: introduce a `trait Rewriter { fn rewrite(..); fn residual_matches(..); }` with a regex impl and a script impl, drop the closures.
- `plan.rs:112-137` vs `151-181` — `plan_rewrite` and `plan_rewrite_scripted` are byte-for-byte identical except for the closures passed into `process_one` and the bool fed to `finalize_plan`. Fix: collapse to one `plan_with_rewriter<R: Rewriter>` that both public entry points wrap.

**Coupling (2)**
- `plan.rs:136, 202-224` — `finalize_plan` takes a `convergent_or_scripted: bool` that overloads two unrelated invariants (static regex convergence vs "trust the dynamic check"). The flag is a leak of caller context into a helper. Fix: pass a `enum AlreadyAppliedPolicy { OnZeroMatches, Never }` (or just `bool already_applied_on_zero`) computed at the call site.
- `plan.rs:293-311 + commit.rs:137-163` — `read_text_or_skip_binary` exists purely to hand `Permissions` to `stage_one` so commit can avoid a second stat; the `permissions` field is the only reason the planner returns metadata. The coupling is the implicit protocol "planner stat must match what apply expects". Fix: either own permission preservation entirely in `commit.rs` (one extra stat in the apply hot path, simpler plan struct) or make the contract explicit via a typed `CapturedMetadata` struct instead of a bare `Permissions`.

**Top targets**
1. `plan.rs:139-141` — drop the per-file regex convergence scan whenever `is_convergent()` is statically true; biggest perf win, removes O(file-size) work per changed file in the common case.
2. `plan.rs:168-171` — replace the second `rewrite_text_scripted` call with a `regex::find_iter().count()`; eliminates a full second pass of user scripts on the hot path of scripted rewrites.
3. `plan.rs:73-81 + 277-285` — stop storing `after` and `diff` side-by-side in `FileChange`; pick one storage and derive the other lazily to cut planner peak memory roughly in half on large rewrites.
4. `plan.rs:208 + 265` — fix the `total_matches` accounting so the `--at-least`/`--at-most` guard sees every match the rewrite engine saw, not only matches in files that survived the no-op filter.
5. `plan.rs:112-137 / 151-181` — collapse the two `plan_rewrite*` entry points behind one `Rewriter` trait; eliminates the duplicated pipeline and removes the `convergent_or_scripted` boolean from `finalize_plan`.
