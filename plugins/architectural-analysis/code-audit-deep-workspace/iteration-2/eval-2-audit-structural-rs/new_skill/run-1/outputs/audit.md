**Perf (4)**
- `structural.rs:225` + `structural.rs:182` — per-match capture lookup is O(captures) and runs once per template part and once per match. Fix: build a small `Vec<Option<&QueryCapture>>` indexed by capture index once per match, then index it.
- `structural.rs:443-450` — friendly path opens a second `Parser`, validates ABI, and parses the substituted pattern; `compile()` then opens a *third* probe parser at `:145` and re-parses the same `Language`. Fix: have the friendly path return a compiled query directly or expose `CompiledStructural::from_query_object` so the friendly probe and the compile probe collapse.
- `structural.rs:182` — fallback "primary capture" picks the numerically-highest capture index across `m.captures` per match, not a meaningful node. For `(a) @x (b) @y` it's an artifact of capture-name ordering. Fix: bind the outermost (or first) capture explicitly during `compile` and store its index next to `root_capture_idx`.
- `structural.rs:432-434` — `structural_rewrite_friendly` re-runs `compile_friendly_query` *and* `CompiledStructural::compile` on every call, which is fine for one-shot use but a footgun if any caller ever loops it. Fix: split into `compile_friendly(...) -> CompiledStructural` + thin `apply` wrapper, mirroring the regex pipeline's compile/scan split.

**Correctness (2)**
- `structural.rs:464` — root-unwrap hard-codes `root.kind() == "source_file"`. Python's root is `module`, Bash is `program`, JSON is `document`, Markdown is `document`. For those grammars the friendly path emits a pattern rooted at the language's whole-file container, so user patterns like `def $NAME(): pass` will not match anywhere. No friendly-pattern tests exist for non-Rust grammars in `structural_tests.rs` to catch this. Fix: keep a small per-`Language` table of root container kinds, or use a structural heuristic (single named child of unnamed root).
- `structural.rs:474-479` — if `emit_node` produces an empty `buf` (e.g., pattern rooted at an unnamed node), `format!("{trimmed} @root")` yields ` @root`, which then errors deep inside `Query::new` with a confusing surface message. Fix: return `Error::StructuralQuery("pattern produced empty query")` when `trimmed.is_empty()`.

**Complexity (3)**
- `structural.rs:513-561` — `emit_node` recurses unbounded over the parsed `--ast` pattern AST; depth equals pattern nesting depth, which is attacker-controlled (the pattern comes from CLI input). Stack-overflow vector on pathological patterns (deeply parenthesised exprs, long generic chains). Sibling `subtree_ellipsis_capture:619-648` already uses an explicit `Vec` stack — adopt the same shape here. Fix: iterative walk with an explicit `Vec<(Node, &'static str)>` worklist; emit open/close tokens off the stack.
- `structural.rs:513-518` — recursive `emit_node` takes 5 mutable / borrowed parameters that all want to live together (`buf`, `predicates`, `lit_counter`, `node`, `src`). Fix: collapse to a `QueryEmitter<'a> { buf, predicates, lit_counter, src }` struct with `fn emit(&mut self, node)`; the iterative rewrite above falls out naturally.
- `structural.rs:168-216` — `CompiledStructural::apply` does match iteration, primary-capture resolution, render, sort, capacity calc, overlap-skip splice, all in one function. Fix: extract `collect_hits` and `splice` so each is one screen, then tests can target overlap-skip without standing up tree-sitter.

**Coupling (3)**
- `structural.rs:341-382` + `plan.rs:183-224` — `plan_structural_rewrite` reimplements three private helpers from `plan.rs` inline: `scan` (lines 348-352), `collect_changes` (lines 364-369), and `finalize_plan` (lines 371-381). Fix: promote `scan`, `collect_changes`, and `finalize_plan` to `pub(crate)` and call them from the structural pipeline; the only structural-specific bit is `convergent_or_scripted = true`.
- `structural.rs:384-408` + `plan.rs:248-286` — `plan_one` and `process_one` are the same shape (`read_text_or_skip_binary` → `rewrite` → if-no-change-return-None → diff → build `FileChange`), but `process_one` is generic over rewrite/converge closures while `plan_one` is hand-rolled. Fix: widen `process_one`'s closure types to also accept `&mut Parser`/`&mut QueryCursor` (or wrap the structural rewrite in a closure that closes over them inside `map_init`) and reuse it.
- `structural.rs:14-17` — `structural.rs` reaches into `plan.rs`'s `pub(crate) read_text_or_skip_binary` and re-exports nothing back; `plan.rs` does not depend on `structural.rs`. The two files form an asymmetric "structural is a second-class citizen of plan" relationship that the import graph hides. Fix: lift `read_text_or_skip_binary` (and the helpers above) into a new `pipeline.rs` so both orchestrators consume from a shared core instead of one importing the other's privates.
