**Perf (3)**
- `structural.rs:184 + 225` — `m.captures.iter().find(...)` runs once per match in `apply` and once per capture-bearing template part in `render`, giving O(captures * template_parts) per match. Fix: build a `[Option<&QueryCapture>; N]` lookup keyed by capture index once per match, index into it.
- `structural.rs:174` — `Parser::parse` runs with no timeout or cancellation flag, so a pathological grammar/source pair can hang a worker thread indefinitely. Fix: set `parser.set_timeout_micros(opts.max_parse_micros)` (plumb via `PlanOptions`) and translate the resulting `None` into a typed `StructuralParse` for that file only.
- `structural.rs:432-434` — `structural_rewrite_friendly` parses the pattern once in `compile_friendly_query`, then `structural_rewrite` re-resolves `ts_language()` and re-sets the parser language in `CompiledStructural::compile`. Fix: have `compile_friendly_query` return the already-resolved query string *and* the `Language` so the second resolve is skipped, or expose a single `compile_friendly` that returns `CompiledStructural` directly.

**Correctness (3)**
- `structural.rs:552` — `node.field_name_for_child(child.id() as u32)` is wrong on two axes: `child.id()` is a pointer-derived `usize` node identifier, not a child index, and the API for the `named_children` iterator is `field_name_for_named_child(named_idx)`. Field constraints (`name:`, `body:`, `parameters:`) are silently dropped from every generated query; matches still happen because tree-sitter falls back to unconstrained shape matching. Fix: track `let mut named_idx = 0u32` across the loop and call `node.field_name_for_named_child(named_idx)`, incrementing per child.
- `structural.rs:281 + 507` — `literal.push(b as char)` / `out.push(b as char)` iterates raw bytes and casts each to `char`, which corrupts every multi-byte UTF-8 codepoint in templates and `--ast` patterns (each continuation byte becomes its own Latin-1 char). Fix: walk by `char_indices()` and copy the codepoint, only switching to byte-level scanning around the ASCII `$` / `{` / `}` sentinels.
- `structural.rs:206-212` — overlap resolution silently swallows later hits that start inside a previous hit's range, with no diagnostic and no count surfaced to the user. The match count returned is `applied`, not the number of matches the query produced, so the user has no way to know matches were dropped. Fix: track and report `skipped_overlaps` in `StructuralOutcome` (and route into the diff summary / JSON output) so callers can see when a pattern is producing overlapping rewrites.

**Complexity (2)**
- `structural.rs:513-561` — `emit_node` recurses on user-controlled pattern AST depth with no bound. A `--ast` pattern of the shape `((((...))))` (or any deep nesting that parses) blows the stack. Fix: convert to an explicit work-stack mirroring `subtree_ellipsis_capture` (619), or guard with a `MAX_DEPTH` constant and return `Error::StructuralQuery("pattern too deeply nested")`.
- `structural.rs:619-648` — `subtree_ellipsis_capture` is invoked on every named subtree from inside `emit_node` and itself does a full DFS of that subtree, so the pattern walk is O(N^2) on pattern node count. For human-sized patterns this is fine, but combined with the recursion-depth issue above the pattern walker is doing more work than its inputs warrant. Fix: hoist a single pre-pass that tags each node with "contains-ellipsis + count" and have `emit_node` read the tag.

**Coupling (4)**
- `structural.rs:348-351 + plan.rs:183-190` — walk-and-`max_files` check is copied verbatim; `plan.rs::scan` already does exactly this. Fix: lift `scan` to `pub(crate)` and call it from both pipelines.
- `structural.rs:364-369 + plan.rs:192-200` — `Vec<Result<Option<FileChange>>>` collation into `Vec<FileChange>` is duplicated. Fix: lift `plan.rs::collect_changes` to `pub(crate)` and call it.
- `structural.rs:371-381 + plan.rs:202-224` — zero-matches → `PlanOutcome::AlreadyApplied`, otherwise `check_match_counts` + `PlanOutcome::Changes`, is reimplemented inline. Fix: call `finalize_plan(changes, true, files_scanned, opts)` (structural is treated as scripted-style "trust the per-file result," so the `convergent_or_scripted` arg is `true`).
- `structural.rs:384-408 + plan.rs:248-286` — `plan_one` and `process_one` share the read → rewrite → build-`FileChange` skeleton but the structural variant skips the convergence callback and inlines the rewrite call. Fix: parameterise `process_one` to accept a `None` convergence check (already plumbed via the `C` generic in plan.rs) so the structural pipeline can pass a noop and reuse the same function.
