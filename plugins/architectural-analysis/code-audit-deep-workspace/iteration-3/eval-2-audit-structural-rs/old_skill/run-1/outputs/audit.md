**Perf (5)**
- `structural.rs:182` — when `root_capture_idx` is `None`, every match recomputes `captures.iter().map(...).max()` to pick the primary. Fix: pick the primary capture once per match via a closure cached on `CompiledStructural` (record "use root idx" vs "use last/max idx" at compile time).
- `structural.rs:184` — `m.captures.iter().find(|c| c.index == primary_idx)` is a second linear scan of the same capture list already scanned on the previous line. Fix: fold "find primary" into the same single pass that picks the index.
- `structural.rs:219` + `:225` — `render` recomputes `template_size_hint()` per match and does an O(captures) `find` for every `TemplatePart::Capture` per match. Fix: store the size hint on `CompiledStructural` at compile time; index captures by position once per match into a small `SmallVec<[Option<&Node>; N]>` keyed by `capture_index`.
- `structural.rs:295` — `push_capture` does a linear `capture_names.iter().position(...)` for every `$NAME` in the template. Fix: build a `HashMap<&str, usize>` once at template-parse entry and look up by name.
- `structural.rs:201` — extra-capacity calc walks `hits` a second time after the `sort_by_key` walk. Fix: accumulate `extra` while pushing hits in the `while let Some(m)` loop (single pass).

**Correctness (3)**
- `structural.rs:174` — `parser.parse(...).ok_or(Error::StructuralParse)` collapses three distinct failure modes (language unset, cancellation, source-length cap) into one opaque variant with no file path. Fix: thread `path` (or at least file label) into the error and distinguish "parser not configured" (a bug, not user error) from "parse aborted".
- `structural.rs:164` — `let _ = parser.set_language(&self.ts_lang)` silently discards the Result on every per-thread parser construction. The comment claims compile-time probe makes it infallible, but the probe uses a *different* `Parser` instance; the invariant is "tree-sitter `set_language` is pure of parser state", which is plausible but undocumented. Fix: propagate the error (return `Result<Parser>`) — `new_parser` runs once per worker, not per file, so the cost is negligible.
- `structural.rs:628` — `subtree_ellipsis_capture` uses `n.utf8_text(src).ok()?`, which bails the whole detection on the first non-UTF8 leaf and silently falls back to the general emit path. The pattern source is `&str` so this is reachable only through grammar quirks, but the silent classification masks a parse-tree anomaly. Fix: treat utf8 failure as a hard `Err` (or at least log via `tracing::warn!`) instead of returning `None`.

**Concurrency (1)**
- `structural.rs:357` — `files.par_iter()` runs on the global rayon pool. If the binary installs a `--threads N` scoped pool only around `plan_rewrite`, the structural pipeline will silently ignore that flag. Fix: ensure the CLI's scoped-pool `install` wraps `plan_structural_rewrite` too (verify in `main.rs`), or accept a `&ThreadPool` parameter.

**Memory (2)**
- `structural.rs:121` — `Hit { start, end, replacement: String }` keeps a fully-rendered `String` per hit even though hits are immediately spliced into `out` in sort order. For large files with many small replacements this doubles peak memory unnecessarily. Fix: render replacements lazily into `out` after sorting `(start, end, captures-snapshot)` tuples, or fold replacements into the splice loop directly.
- `structural.rs:303` — `TemplatePart::Capture { index, name: String }` stores the capture name solely to render error messages on lookup failure (line 226). The name can be looked up from `query.capture_names()[index]` on the rare error path. Fix: drop the `name: String` field; resolve via `capture_names` only when constructing the error.

**Complexity (4)**
- `structural.rs:168-216` — `CompiledStructural::apply` is ~50 lines doing three jobs (collect hits, sort, splice). Fix: split into `collect_hits(&self, parser, cursor, source) -> Result<Vec<Hit>>` and `splice(source, hits) -> (String, usize)`.
- `structural.rs:440-480` — `compile_friendly_query` is ~40 lines mixing metavar substitution, parse, error-shape massaging, `source_file` unwrap, and final formatting. Fix: extract `parse_pattern(&ts_lang, &substituted) -> Result<Node>` and `format_query(buf, predicates) -> String`.
- `structural.rs:513-561` — `emit_node` is an unbounded recursive walker over a tree-sitter parse tree (the user-supplied `--ast` pattern). A deeply nested pattern would blow the Rust stack. The function is also doing four distinct things (wildcard ellipsis, single-node metavar, literal-leaf with `#eq?`, recursive named-children emit) — borderline god-function. Fix: rewrite as an explicit `Vec<Node>` stack walk like `subtree_ellipsis_capture` already does, and split the four arms into named helpers.
- `structural.rs:619-647` — `subtree_ellipsis_capture` uses an explicit stack (good) but discards field-name structure during the walk; this is fine for the ellipsis-count predicate but worth a one-line comment noting that the walk intentionally ignores field positions. Fix: short `// why` comment.

**Coupling (3)**
- `structural.rs:348-381 + plan.rs:183-224` — `plan_structural_rewrite` reimplements `scan`, `collect_changes`, and most of `finalize_plan` inline instead of calling the helpers `plan.rs` already exposes for exactly this purpose (`scan` is `fn`-private, `collect_changes` and `finalize_plan` are also private). Fix: make those three helpers `pub(crate)` and reuse them; the structural path's only divergence is "treat zero matches as `AlreadyApplied` unconditionally", which is one boolean flag on `finalize_plan` (already parameterized as `convergent_or_scripted`).
- `structural.rs:384-408 + plan.rs:248-286` — `plan_one` (structural) and `process_one` (regex/script) are ~70% identical: read-or-skip, run rewrite, compare text, build label+diff, construct `FileChange`. The differences are (a) no convergence check in structural, (b) outcome shape. Fix: extract `build_file_change(path, before, after, matches, permissions) -> FileChange` into `plan.rs` and call from both.
- `structural.rs:14-17 + plan.rs:293` — `read_text_or_skip_binary` is `pub(crate)` in `plan.rs` and imported from there into `structural.rs`. Its docstring already says "Shared by the regex and structural pipelines" — meaning the helper has outgrown `plan.rs`. Fix: lift `read_text_or_skip_binary` (and the proposed `build_file_change` above) into a small new `plan_common.rs` or into `walker.rs` so neither orchestrator imports private internals of the other.
