**Perf (4)**
- `plan.rs:140` — convergence check uses `find_iter(after).count()`, full O(|after|) scan when we only branch on `extra > 0`. Fix: use `regex.is_match(after)` and report `extra: 1` (or drop the count from the error) for a first-match short-circuit.
- `plan.rs:271` — per-file convergence re-runs the whole regex over the post-image even when `compiled.is_convergent()` already proved the static template can't re-match. Fix: still required for boundary effects, but gate the dynamic check behind a cheap pre-filter (e.g., only re-scan files where a match touched the boundary of replacement text) or at least share work with the next finding.
- `plan.rs:264-265` — `outcome.after == before` is an O(n) equality check done after every successful rewrite, even though `matches == 0` already covers the no-op case for non-pathological templates. Fix: drop the equality branch and rely on `matches == 0` plus a single byte-length cheap check; only fall through to full compare when lengths match exactly.
- `plan.rs:123,162` — `par_iter().map(...).collect::<Vec<Result<Option<FileChange>>>>()` materializes a slot per scanned file (≥sizeof(FileChange) each) before the serial drain. Fix: `collect::<Result<Vec<_>>>()` over `filter_map` so Nones never enter the vector and a single failure short-circuits the join.

**Memory (3)**
- `plan.rs:73-81` — `FileChange` holds both the full post-image `after` and the rendered unified `diff`; for change-heavy files the diff approaches the size of `before + after`, so the planner's peak resident set is roughly 2× the rewritten content even though `apply_changes` only reads `after` (verified: `diff` is consumed only by `recast/src/main.rs:446`). Fix: render `diff` lazily (closure or `Option<String>` filled on demand) so `--apply` runs don't carry it, or store the pre-image and re-render on read.
- `plan.rs:259-285` — `before` and `outcome.after` and the freshly built `diff` are all live simultaneously inside `process_one`; `similar::TextDiff::from_lines` additionally tokenizes both sides, so peak per-file footprint is `|before| + |after| + 2× line-index + |diff|`. Fix: drop `before` immediately after `unified_diff` returns by scoping it inside an inner block, or push diff rendering out of the hot path (see previous finding).
- `plan.rs:120,159` — `scan` materializes the entire `Vec<PathBuf>` before any rewrite work begins. Required for `par_iter`, but for very large trees the path list itself becomes large. Fix: stream into a bounded channel feeding a rayon scope, or chunk-process via `par_bridge` so peak path-list memory is bounded.

**Complexity (2)**
- `plan.rs:248-286` — `process_one` is parameterised on two closures (`rewrite`, `convergence_check`) that always co-vary (regex pair or scripted pair); the resulting `where R: Fn..., C: Fn...` signature obscures that there are only two real call shapes. Fix: replace the two closures with one `trait Rewriter { fn rewrite(...); fn convergence(...); }` and dispatch through `&dyn Rewriter` or two impls.
- `plan.rs:202-224` — `finalize_plan` takes a flag named `convergent_or_scripted` that means "convergent (regex) OR scripted (trust dynamic check)". Two unrelated facts collapsed into one bool. Fix: pass an enum (`ConvergenceProof::Static(bool) | ConvergenceProof::Dynamic`) so the call sites document intent and the branch in `finalize_plan` reads correctly.

**Coupling (2)**
- `plan.rs:112-137 + plan.rs:151-181` — `plan_rewrite` and `plan_rewrite_scripted` differ only in the two closures passed to `process_one` and the final `convergent_or_scripted` argument; rest of the pipeline (compile, scan, parallel map, collect, finalize) is byte-identical. Fix: extract a private `run_plan(...)` that takes the rewrite/convergence closures plus the convergence proof, and have both public entry points call it.
- `plan.rs:120-137 + structural.rs:348-381` — `plan_structural_rewrite` re-implements the same scan → parallel-rewrite → collect → finalize pipeline as `plan_rewrite`, including its own duplicated max-files check (`structural.rs:349`), its own Nones-included `Vec<Result<Option<FileChange>>>` collect, and its own zero-matches / `check_match_counts` epilogue. Three orchestrators of the same recipe. Fix: route structural through the same internal `run_plan` and supply a `Rewriter` impl that owns a `Parser`/`QueryCursor`; delete the per-call boilerplate.

**Top targets**
1. `plan.rs:73-81 / 277-278` — drop `diff` from `FileChange` (or make it lazy). Biggest single memory win; the field is dead weight on every `--apply` run and is the only thing keeping per-change peak at 2× content size.
2. `plan.rs:140` — replace the `find_iter().count()` convergence probe with `is_match()`. O(|after|) → first-match short-circuit on every changed file, no semantic change to the success path.
3. `plan.rs:123,162` — collect into `Result<Vec<FileChange>>` via `filter_map` instead of `Vec<Result<Option<FileChange>>>`. Removes a per-file slot the size of a full `FileChange` for every scanned (not changed) file.
4. `plan.rs:112-181 + structural.rs:348-381` — collapse the three near-identical orchestrators into one internal `run_plan`. Eliminates the duplication that lets future bugs (e.g., the missing structural convergence check, the missing `at_least=0` early return parity) diverge silently.
5. `plan.rs:248-286 + 202-224` — replace the closure-pair + `convergent_or_scripted` bool with a small `Rewriter` trait + `ConvergenceProof` enum. Makes the new unified `run_plan` readable and gives scripted/structural a typed seam to extend.
